### 2. Skill 定位

god-view 是三个基础 skills (project-init, project-sync, project-review) 之上的**战略层工具**：

```
              ┌─────────────┐
              │  god-view   │  ← 战略层：识别主要矛盾，提供方向
              └─────────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
   ┌────▼────┐  ┌───▼────┐  ┌───▼──────┐
   │ project │  │project │  │  project │  ← 执行层：初始化、同步、审查
   │  init   │  │  sync  │  │  review  │
   └─────────┘  └────────┘  └──────────┘
```

### 3. 核心方法论

god-view 融合三种思想：

#### 3.1 毛泽东 - 主要矛盾论
- 识别主要矛盾和次要矛盾
- 使用四象限法（影响范围、阻塞关系、风险累积、解决杠杆）
- 主要矛盾的转化规律

#### 3.2 马斯克 - 第一性原理
- 回归根本目标
- 质疑一切"应该"
- 从零推导，不依赖惯例

#### 3.3 乔布斯 - 极简主义
- 每个元素都必须有存在理由
- 信噪比 > 80%
- Less is more

### 4. Skill 功能

**god-view 做的事**：
- ✅ 识别项目当前的主要矛盾（1-3 个）
- ✅ 从第一性原理审视项目根本目标
- ✅ 极简主义审查（找出冗余和必要）
- ✅ 提供高杠杆的战略方案
- ✅ 推荐下一步应执行哪个 skill

**god-view 不做的事**：
- ❌ 不执行具体文档更新（交给 project-sync）
- ❌ 不执行详细健康检查（交给 project-review）
- ❌ 不执行项目初始化（交给 project-init）
- ❌ 不修改代码或文档

### 5. 使用场景

**适合使用 god-view**：
- 重大决策前（架构重构、技术选型）
- 感觉迷茫时（问题太多，不知从何入手）
- 资源有限时（必须聚焦最关键问题）
- 项目陷入困境时（进展缓慢、技术债累积）
- 周期性战略审查（每季度/半年）

**不适合使用 god-view**：
- 日常功能完成后的文档同步 → 用 project-sync
- 定期文档健康度检查 → 用 project-review
- 项目初次建立知识库 → 用 project-init

### 6. 输出报告结构

god-view 产出 `God-View Analysis Report`，包含：

1. **战略摘要**
   - 主要矛盾识别（1-3 个）
   - 如果只能做一件事的建议

2. **第一性原理分析**
   - 根本目标 vs 当前状态
   - 必要的 vs 多余的

3. **极简主义审查**
   - 违反极简主义的地方
   - 信噪比评估

4. **矛盾动态**
   - 当前阶段判断
   - 矛盾转化预测

5. **战略方案**
   - 如果只能做一件事
   - 最小可行方案
   - 推荐方案
   - 扩展方案

6. **执行路径**
   - 下一步应调用哪个 skill
   - 如何验证效果

7. **自检框架**
   - 给用户的思考框架

## 核心公式

```
God-View = First Principles (找到根本)
         + Theory of Contradictions (抓住重点)
         + Minimalism (删繁就简)
```

## 三条铁律

1. **永远从第一性原理出发** - 质疑每一个"应该"，回归根本需求
2. **永远识别主要矛盾** - 不要平均用力，聚焦高杠杆问题
3. **永远追求极简** - 每一个元素都必须有存在理由

## 最终目标

> 用最少的资源，解决最核心的问题，创造最大的价值
