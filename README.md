# Project Skills - AI 可读的项目知识管理工具集

一套通用的项目知识管理 Skills，适用于所有支持 AGENTS.md 规范的 AI 工具（如 Claude Code、Cursor、Windsurf 等），帮助 AI 和人类更好地理解、维护和演进项目。

## 核心理念

**用最少的文档，解决最核心的问题，创造最大的价值**

本工具集融合三种思想：
- **毛泽东的主要矛盾论** - 识别核心问题，区分主次
- **马斯克的第一性原理** - 回归根本，质疑惯例
- **乔布斯的极简主义** - 删繁就简，追求本质

## Skills 说明

### 🎯 战略层

#### god-view (上帝视角)
> 战略分析工具，用于重大决策前的全局审视

**适用场景**：
- 重大决策前（架构重构、技术选型）
- 感觉迷茫时（问题太多，不知从何入手）
- 资源有限时（时间紧、人手少）
- 项目陷入困境
- 周期性战略审查（每季度/半年）

**核心功能**：
- 识别主要矛盾（1-3 个核心问题）
- 第一性原理分析（根本目标 vs 当前状态）
- 极简主义审查（找出冗余和必要）
- 提供高杠杆解决方案
- 推荐下一步执行路径

**不做的事**：
- 不执行具体文档更新
- 不执行详细健康检查
- 不修改代码或文档

### ⚙️ 执行层

#### project-init (项目初始化)
> 为项目建立 AI 可读的知识库基线

**适用场景**：
- 首次接手新仓库或已有项目
- 缺少 AGENTS.md 或 docs/ai-context
- 现有文档零散/过期
- 准备长期维护/重构/大功能前

**创建的知识结构**：
```
AGENTS.md              # 项目工作协议
CLAUDE.md              # 符号链接（Claude Code 识别用）
docs/ai-context/
  ├── architecture.md  # 架构说明
  ├── modules.md       # 模块说明
  ├── data-flow.md     # 数据流
  ├── database.md      # 数据库
  ├── api.md           # API 说明
  ├── features.md      # 需求/功能/优化点
  └── decisions/       # 决策记录
```

#### project-sync (项目同步)
> 保持项目现实和知识库同步

**适用场景**：
- 功能完成后
- Bug fix 或重构改变重要行为后
- 数据库/API/集成发生变化后
- Commit 前或 PR 前

**功能**：
- 基于 git diff 判断影响范围
- 只更新受影响的文档
- 默认提醒低风险技术债
- 维护决策记录

#### project-review (项目审查)
> 周期性检查知识健康度

**适用场景**：
- 项目持续变化一段时间后
- 计划较大功能/重构/架构调整前
- 新成员或新 AI Agent 加入前
- 作为周期性项目维护
- 多次执行 project-sync 后检查漂移

**检查内容**：
- 文档和代码是否一致
- 哪些知识缺失
- 哪些描述过期或误导
- 技术债情况
- features.md 是否仍有必要

## 快速开始

### 场景 1：新项目或首次使用

**在 Claude Code 中**：
```bash
# 调用 project-init 建立知识库
/project-init

# 查看生成的文档
ls AGENTS.md docs/ai-context/
```

**在其他 AI 工具中**：
- 如果工具支持自定义 skills/agents，导入对应的 SKILL.md
- 如果工具只识别 AGENTS.md，可以手动参考本工具集的指令来指导 AI
- 参考 `docs/ai-context/core-methodology.md` 了解核心方法论

### 场景 2：日常开发后同步

**在 Claude Code 中**：
```bash
# 完成功能开发后
/project-sync

# 准备 commit 前
/project-sync
```

**在其他 AI 工具中**：
- 告诉 AI："按照 project-sync 的方式同步项目知识"
- 或直接说："基于 git diff 更新 docs/ai-context/ 中受影响的文档"

### 场景 3：感觉迷茫或重大决策前

**在 Claude Code 中**：
```bash
# 1. 先用 god-view 战略分析
/god-view

# 2. 根据 god-view 的建议，调用对应 skill
/project-init   # 如果建议初始化知识库
/project-sync   # 如果建议同步文档
/project-review # 如果建议审查健康度
```

**在其他 AI 工具中**：
- 告诉 AI："用上帝视角分析项目，识别主要矛盾"
- 或直接说："用第一性原理+主要矛盾论+极简主义分析项目"

### 场景 4：周期性健康检查

**在 Claude Code 中**：
```bash
# 每月或每季度
/project-review

# 如果发现问题，执行推荐的修复
/project-sync
```

**在其他 AI 工具中**：
- 告诉 AI："审查项目知识健康度，检查文档是否匹配代码"

## god-view 的三层分析法

### 第一层：第一性原理分析
**找到根本目标**

关键追问：
- 项目的根本目标是什么？
- 什么知识是必须的？
- 什么知识是多余的？
- 如果今天重新开始，会怎么做？

### 第二层：主要矛盾分析
**找到核心问题**

四象限法判断主次矛盾：

| 维度 | 主要矛盾 | 次要矛盾 |
|------|---------|---------|
| 影响范围 | 系统性、全局性 | 局部、模块级 |
| 阻塞关系 | 阻塞其他工作 | 不阻塞其他工作 |
| 风险累积 | 复利型恶化 | 一次性、不恶化 |
| 解决杠杆 | 解决后带动多个问题 | 只解决自身 |

判断方法：
```
如果只能解决一个问题，解决哪个能让项目状态改善最多？
→ 这就是主要矛盾
```

### 第三层：极简主义过滤
**删繁就简**

过滤原则：
- 每个文档必须回答：存在理由？不存在的后果？
- 每句话必须回答：传递什么信息？读者损失是什么？
- 目标：信噪比 > 80%

## 方法论示例

### 示例 1：识别主要矛盾

**场景**：项目有 20 个问题，不知从何入手

❌ **错误做法**（平均用力）：
```
发现 20 个问题，建议逐个解决
```

✅ **正确做法**（识别主要矛盾）：
```
主要矛盾：数据库 schema 变更频繁但无文档，
导致前端团队频繁误判数据结构

为什么是主要矛盾：
- 影响范围：系统性（前后端协作）
- 阻塞关系：每次开发都要反复确认
- 风险累积：复利型（越积越严重）
- 解决杠杆：建立后 API 文档也更容易

如果只能做一件事：建立 database.md

其他 19 个问题都是次要矛盾
```

### 示例 2：第一性原理思考

**场景**：按惯例应该有很多标准文档

❌ **错误做法**（基于惯例）：
```
应该创建：
- README
- CONTRIBUTING
- CODE_OF_CONDUCT
- CHANGELOG
- LICENSE
...
```

✅ **正确做法**（第一性原理）：
```
项目知识的根本目的：让新成员 1 小时内开始贡献

必须回答的问题：
1. 项目干什么？→ README（保留）
2. 如何运行测试？→ README（保留）
3. 代码如何组织？→ architecture.md（保留）

不必要的：
- CODE_OF_CONDUCT：3 人小团队，面对面沟通
- CONTRIBUTING：内部项目，口头约定
- CHANGELOG：git log 足够

最小集合：README + architecture.md + decisions/
```

### 示例 3：极简主义审查

**场景**：项目太复杂，难以理解

❌ **错误做法**（只指出问题）：
```
发现：
- 抽象层级过多
- 模块边界不清
- 文档和代码不匹配
```

✅ **正确做法**（极简审查 + 方案）：
```
违反极简主义的核心问题：
1. 抽象层级过多（从 API 到数据库经过 5 次转换）
   简化建议：合并为 2 层（API + Repository）

信噪比评估：
- 当前：40%（10k 行代码，4k 核心逻辑）
- 主要噪音：过度抽象、重复逻辑

如果只能做一件事：合并抽象层级，从 5 层到 2 层
```

## 三条铁律

1. **永远从第一性原理出发**  
   质疑每一个"应该"，回归根本需求

2. **永远识别主要矛盾**  
   不要平均用力，聚焦高杠杆问题

3. **永远追求极简**  
   每一个元素都必须有存在理由

## 核心公式

```
好的项目知识管理 = 第一性原理（找根本）
                  + 主要矛盾论（抓重点）
                  + 极简主义（删冗余）
```

## 兼容性

本工具集遵循 AGENTS.md 规范，兼容所有支持该规范的 AI 开发工具：

### ✅ 已验证支持
- **Claude Code** - Anthropic 官方 CLI 工具
- **Cursor** - AI-first 代码编辑器
- **Windsurf** - AI 协作开发环境

### 📋 理论支持
任何识别 `AGENTS.md` 文件作为项目指令的 AI 工具都可以使用本工具集。

### 🔧 如何使用

本工具集提供双文件名支持：
- **AGENTS.md** - 主文件，供通用 AI 工具识别
- **CLAUDE.md** - 符号链接指向 AGENTS.md，供 Claude Code 识别

在你自己的项目中使用时：
1. 创建 `AGENTS.md` 作为主文件
2. 可选：创建符号链接 `CLAUDE.md -> AGENTS.md`（如果使用 Claude Code）
3. 按照本工具集的知识结构组织 `docs/ai-context/`

## 与其他工具的区别

### vs 传统文档
- **传统文档**：全面、详细、容易过期
- **本工具集**：精简、必要、随代码演进

### vs README-driven
- **README-driven**：从文档开始设计
- **本工具集**：从代码事实建立知识，且持续同步

### vs 自动生成文档
- **自动生成**：从代码生成，但无法捕捉决策背景
- **本工具集**：记录代码无法自解释的上下文

### vs 其他 AI 项目文档工具
- **本工具集特点**：
  - ✅ 通用规范（AGENTS.md），不绑定特定 AI 工具
  - ✅ 融合方法论（第一性原理+主要矛盾论+极简主义）
  - ✅ 战略+执行双层架构（god-view + init/sync/review）
  - ✅ 强调知识同步，而非一次性生成

## 最佳实践

### DO ✅

- ✅ 每次有意义变更后调用 project-sync
- ✅ 每月或每季度调用 project-review
- ✅ 遇到重大决策时先调用 god-view
- ✅ 保持文档精简，信噪比 > 80%
- ✅ 明确区分：事实、建议、决策
- ✅ 从第一性原理思考

### DON'T ❌

- ❌ 不要为了"完整"创建不必要的文档
- ❌ 不要把推测写成事实
- ❌ 不要平均用力解决所有问题
- ❌ 不要忽略主要矛盾去解决次要问题
- ❌ 不要让文档和代码产生漂移
- ❌ 不要基于惯例而不思考根本需求

## 常见问题

### Q: 只能在 Claude Code 中使用吗？
A: 不是。本工具集遵循 AGENTS.md 规范，可以在任何支持该规范的 AI 工具中使用（如 Cursor、Windsurf 等）。Claude Code 的 `/skill` 命令只是一种调用方式，你也可以直接告诉其他 AI 工具按照这些指令工作。

### Q: 如何在 Cursor/Windsurf 等工具中使用？
A: 有两种方式：
1. **如果工具支持自定义 skills**：导入对应的 SKILL.md 文件
2. **如果工具只识别 AGENTS.md**：在你的项目 AGENTS.md 中引用本工具集的方法论和指令，然后直接对话告诉 AI 要做什么（如"用上帝视角分析项目"）

### Q: AGENTS.md 和 CLAUDE.md 有什么区别？
A: AGENTS.md 是通用规范，被多种 AI 工具支持。CLAUDE.md 是 Claude Code 的专属文件名。本项目中 CLAUDE.md 是指向 AGENTS.md 的符号链接，确保两种工具都能识别。

### Q: 和 project-init/sync/review 有什么区别？
A: god-view 是战略层工具，只识别问题和提供方向，不执行具体操作。project-init/sync/review 是执行层工具，负责具体的文档创建、更新和审查。

### Q: 什么时候用 god-view？
A: 当你感觉迷茫、面临重大决策、不知道从何入手、或需要周期性战略审查时。

### Q: 必须按顺序使用吗？
A: 不必须。根据实际需求：
- 新项目 → project-init
- 日常变更 → project-sync
- 健康检查 → project-review
- 战略分析 → god-view

### Q: 会创建很多文档吗？
A: 不会。遵循极简主义，只创建必要的文档。默认结构是最小知识集合，不适用的部分会标记为 "Not applicable"。

### Q: 支持什么语言/框架？
A: 通用工具，不绑定任何语言或框架。适用于任何项目。

### Q: 如何验证效果？
A: 
- 新成员是否能在 1 小时内开始贡献代码？
- AI 是否能准确理解项目？
- 是否减少了重复提问？
- 是否降低了协作成本？
