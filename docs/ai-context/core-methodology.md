3. 如果不能清晰回答，考虑是否真的需要这个文档
```

### 4.2 project-sync

**融合后的增强**：

1. **第一性原理应用**：
   - 同步的根本目的：保持文档和代码一致
   - 只同步受影响的部分，不要"顺手"改无关内容
   
2. **主要矛盾识别**：
   - 判断本次变更的主要影响是什么
   - 区分：系统性变化 vs 局部变化
   - 策略：系统性变化优先同步架构文档，局部变化只更新相关模块文档
   
3. **极简主义**：
   - 同步时删除过期内容，而不是累加
   - 每次同步都是一次精简的机会

**新增指令**：
```
在同步前，必须识别：
1. 本次变更的主要矛盾是什么？（架构变化？模块边界调整？局部优化？）
2. 哪些文档受主要矛盾影响？（优先同步）
3. 哪些文档受次要影响？（可延后）
4. 同步时能否删除过期或冗余内容？
```

### 4.3 project-review

**融合后的增强**：

1. **第一性原理应用**：
   - 审查的根本目的：发现阻碍项目演进的核心问题
   - 不要列出所有问题，而是识别主要矛盾
   
2. **主要矛盾识别**：
   - 使用四象限法分类所有发现
   - 明确指出：当前项目的主要矛盾是什么
   - 给出：解决主要矛盾的优先级建议
   
3. **极简主义**：
   - 报告本身要精简：重点突出，而不是面面俱到
   - 建议要具体、可执行，而不是泛泛而谈

**新增指令**：
```
审查报告必须包含：
1. 主要矛盾识别：当前最阻碍项目的核心问题（1-3个）
2. 次要问题清单：其他发现（可简要列出）
3. 矛盾分析：为什么这些是主要矛盾？（影响范围、阻塞关系、风险累积）
4. 解决策略：先解决哪个，为什么，预期效果
5. 极简建议：如果只能做一件事，应该做什么？
```

## 五、判断标准和示例

### 5.1 好的知识管理（符合方法论）

**示例 1：识别主要矛盾**

❌ **不好**（列举所有问题）：
```
发现以下问题：
- README 过时
- 3 个函数命名不一致
- 缺少 API 文档
- 数据库文档缺失
- 5 个 TODO 没有 owner
- ...（共 20 个问题）

建议：逐个解决这些问题
```

✅ **好**（识别主要矛盾）：
```
主要矛盾：数据库 schema 变更频繁，但没有文档，导致前端团队频繁误判数据结构

影响分析：
- 阻塞关系：每次前端开发都要反复确认字段
- 风险累积：复利型，每次变更都会放大问题
- 影响范围：系统性，影响前后端协作

其他发现：
- README 过时（局部问题，不阻塞）
- 命名不一致（次要矛盾）

建议：优先建立 database.md 和 API contract 文档，其他问题延后
```

**示例 2：第一性原理思考**

❌ **不好**（基于惯例）：
```
项目缺少文档，应该创建：
- README
- CONTRIBUTING
- CODE_OF_CONDUCT
- CHANGELOG
- API.md
- ARCHITECTURE.md
- ...
```

✅ **好**（从根本需求出发）：
```
项目知识的根本目的：让新成员能在 1 小时内开始贡献代码

必须回答的问题：
1. 这个项目是干什么的？→ README（保留）
2. 如何运行和测试？→ README（保留）
3. 代码如何组织的？→ architecture.md（保留）
4. 重要决策是什么？→ decisions/（保留）

不必要的：
- CODE_OF_CONDUCT：小团队，不需要
- CONTRIBUTING：团队内部项目，面对面沟通更有效
- CHANGELOG：git log 已足够

最小知识集合：README + architecture.md + decisions/
```

**示例 3：极简主义过滤**

❌ **不好**（冗余描述）：
```
## User Module

The User Module is responsible for managing users in the system. 
It handles user creation, user updates, user deletion, and user queries.
The module is located in src/modules/user/.
It uses the User model defined in src/models/user.ts.
It exposes a UserService class.
The UserService depends on the Database service.
```

✅ **好**（精简到本质）：
```
## User Module

**职责**：用户 CRUD
**位置**：src/modules/user/
**依赖**：Database service
**对外接口**：UserService
```

### 5.2 坏的知识管理（违反方法论）

**反模式 1：平均用力**
- 所有问题一视同仁
- 没有识别主要矛盾
- 试图同时解决所有问题

**反模式 2：过度设计**
- 创建过多文档
- 为了"完整"而添加不必要的章节
- 违反极简主义

**反模式 3：表面文章**
- 只解决表面问题（改命名、调格式）
- 忽略根本问题（架构混乱、职责不清）
- 没有识别主要矛盾

**反模式 4：惯性思维**
- "别的项目都有 CONTRIBUTING，我们也要有"
- "文档应该包含历史、背景、未来规划..."
- 没有从第一性原理思考

## 七、总结

### 核心公式

```
好的项目知识管理 = 第一性原理（找到根本） 
                  + 主要矛盾论（抓住重点）
                  + 极简主义（删繁就简）
```

### 三条铁律

1. **永远从第一性原理出发**：质疑每一个"应该"，回归根本需求
2. **永远识别主要矛盾**：不要平均用力，聚焦高杠杆问题
3. **永远追求极简**：每一个元素都必须有存在理由

### 最终目标

> 用最少的文档，解决最核心的问题，创造最大的价值
